## 1. Primary Benchmarks

### 1.1 HumanEval

| Property | Value |
|----------|-------|
| Problems | 164 |
| Metric | pass@1 (greedy), pass@k (sampling) |
| Format | Function signature + docstring → completion |
| Execution | Python sandbox with test assertions |
| Target | ≥ 30% pass@1 |

Each problem provides a function signature with docstring. The model generates
the function body. Correctness is verified by executing hidden test cases.

**pass@k formula** (unbiased estimator):
```
pass@k = 1 - C(n-c, k) / C(n, k)

n = total samples generated
c = number of correct samples
k = number of attempts allowed
```

### 1.2 MBPP (Mostly Basic Python Problems)

| Property | Value |
|----------|-------|
| Problems | 974 (sanitized subset: 427) |
| Metric | pass@1 |
| Format | Natural language description → code |
| Execution | Python sandbox with 3 test assertions per problem |
| Target | ≥ 40% pass@1 |

MBPP is typically easier than HumanEval — more basic Python operations,
shorter solutions. Good for measuring general Python fluency.

| Inference speed (GPU f16) | < 10 ms/tok | `apr bench --device gpu` |
| Model size (Q4) | < 200 MB | `apr quantize --bits 4` |

### 2.1 val_ppl Tracking

Same-distribution validation perplexity on held-out codeparrot-clean data.
Primary signal for training convergence (NOT for benchmark prediction).

**Critical**: val and train data MUST come from same source. Cross-distribution
val_ppl is misleading. See v6 post-mortem (val/train mismatch bug).

Current best: **776** (v6, step 2000, 64M tokens).
Target post-distillation: **< 100**.

## 3. Evaluation Protocol

### 3.1 Running Evaluations

```bash
# HumanEval greedy (pass@1)
apr eval --task humaneval --model checkpoints/albor-distill/ \
         --temperature 0.0 --max-tokens 512

# HumanEval sampling (pass@k)
apr eval --task humaneval --model checkpoints/albor-distill/ \
         --temperature 0.8 --top-p 0.95 --k 1,5,10 --n 200

# MBPP
apr eval --task mbpp --model checkpoints/albor-distill/ \
         --temperature 0.0 --max-tokens 512

# Quick perplexity check
apr eval --task perplexity --model checkpoints/albor-distill/ \
         --data data/pretokenized-1024-v3/val/
```

### 3.2 Evaluation During Training

Auto-eval every `eval_interval` steps (250 by default). ALB-087 feature:
- Run val_ppl on held-out data
- Early stopping if no improvement for `patience` evals (10 × 250 = 2500 steps)
- Best checkpoint saved separately

### 3.3 Multi-Sample pass@k (ALB-088)

For reliable pass@k estimates:
- Generate n=200 samples per problem at temperature=0.8
- Compute unbiased pass@k for k=1,5,10
- Report mean and 95% confidence interval
- Total generations: 164 × 200 = 32,800 for HumanEval

## 4. Generation Config for Eval

### 4.1 Greedy Decoding

| Parameter | Value |
|-----------|-------|
| temperature | 0.0 |
| max_tokens | 512 |
| stop_sequences | `["\nclass ", "\ndef ", "\n#", "\nif __name__"]` |

Used for pass@1. Deterministic — same model always produces same output.

### 4.2 Sampling

| Parameter | Value |
|-----------|-------|
| temperature | 0.8 |
| top_p | 0.95 |
| max_tokens | 512 |
| stop_sequences | Same as greedy |

Used for pass@k with k>1. Higher temperature produces more diverse samples,
improving pass@k estimates.

## 5. Evaluation Infrastructure

### 5.1 Execution Sandbox

Code generated by the model is executed in a sandboxed environment:
- **Timeout**: 10 seconds per test case
- **Memory limit**: 256 MB
- **No network access**
- **Restricted imports**: Standard library only
- **Process isolation**: Separate process per execution

### 5.2 GPU-Accelerated Inference (ALB-089)

Status: DOGFOODING. CUDA forward pass for fast eval generation.
Without this, eval on CPU is slow (~50ms/tok) — generating 200 samples
per problem takes hours.

With GPU inference: ~5ms/tok → 32,800 × 512 × 5ms ≈ 1.4 hours for full
HumanEval pass@k evaluation.

## 6. Baseline Comparisons

### 6.1 Reference Models at Similar Scale

| Model | Params | HumanEval | MBPP | Training Data |
|-------|--------|-----------|------|---------------|
| phi-1-small | 350M | 45% | 55% | 7B curated |
| CodeGen-350M | 350M | 12% | — | 577B raw |
| SantaCoder | 1.1B | 14% | — | The Stack |
| InCoder-1.3B | 1.3B | 8% | — | 159B tokens |
| Albor (current) | 370M | 0% | — | 64M raw |
| **Albor (target)** | **370M** | **≥30%** | **≥40%** | **~2B curated + distill** |

### 6.2 Key Observation

phi-1-small proves 45% HumanEval is achievable at 350M params. CodeGen-350M
at 12% with 577B raw tokens shows that data quality matters 80x more than
quantity. Our path: match phi-1's data quality through distillation.
